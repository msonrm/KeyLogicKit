# KIDE / GiDE 実装ノート

> CLAUDE.md の参照節から移設（2026-07-06）。KIDE の実装状況・コンポーネント構成の詳細。仕様は `docs/kide-spec.md` / `GiDE-spec.md` を参照。

## GiDE（MVP 打ち切り・ソース削除済み）

**Android 実装 `android-gide/` とビルドワークフロー `gide-android-build.yml` は 2026-07-06 に削除**（Phase 2 軽量版）。gide は GIME(gamepad)↔KIDE(HID) の橋渡し役だったが、両現行アプリが直接コードをほぼ共有しないため共有 Gradle モジュール化はせず、**gide 側にのみ存在した移植漏れバグ修正（RS↓ チャタリング debounce `rsDownDebounceMs`）を GIME (`android/`) へ救出**したうえで削除した。実コードは git 履歴を参照。設計・打ち切り経緯は以下のアーカイブ文書に残す。

- `GiDE-spec.md` — GiDE (Gamepad Interface Device Emulator) MVP スペック。Android を BT HID キーボードに化けさせて任意機器に入力する派生アプリ。**Phase 3 実機検証で「ゲームパッド入力の構造的限界（chord 判定 + ローマ字展開 + 多段イベント）で物理キーボード比 2-4x オーバーヘッドが取れない」と判明し、MVP を打ち切り。後継として KIDE（USB OTG キーボード入力版）に方針転換**
- `docs/gide-spec-eval.md` — GiDE MVP スペックの実現可能性評価メモ（Phase 0.5 spike の追加、ローマ字送出戦略、派生案件 KIDE のアイデア）
- 旧構成（削除前）: `com.msonrm.gide` namespace、HID 送信パイプライン + GIME 入力流用 + `KanaToRomajiTable`。`hid/` は KIDE に流用済み（下記）。GIME (`android/`) とはバージョン独立で、タグ prefix `gide-android-v*` / versionCode 環境変数 `GIDE_VERSION_CODE` を使用していた（`GIDE_PAT` secret は未使用化）

## KIDE（現行）

- `docs/kide-spec.md` — KIDE (Keyboard Interface Device Emulator) MVP スペック。USB OTG 接続の物理キーボード入力を BT HID Device 役で任意機器に転送し、配列変換（大西/Dvorak/AZIK/月配列/薙刀式/親指シフト等）を間に挟む。ペルソナは HHKB + MacBook 尊師スタイルのパワーユーザー or 軽量タブレットノマドワーカーで、「かえうち」というハードウェアが古く感じる層。USB-C 入力 + BT 出力で「かえうち」の機能上位互換を目指す。**Phase 0-3 完遂** (シンプル置換 + AZIK ローマ字出力)、**Phase 4 核仮説 validation 完了** (PR #588 #594 #595 #598): JSON ローダー基盤 + JIS かな直接出力 + 同時打鍵 chord (NICOLA) + 前置/後置シフト (月配列 2-263) を iPad の「日本語 - かな入力」モードで実機動作確認済み。Phase 3 で問題視した iPad ローマ字 IME のラグは JIS かなモードでは構造的に発生せず (kana を直接受け取るので変換処理が無い)、ペルソナの **iPad ペルソナ向け解** が成立。**Phase 4 後半完遂**: 3 キー chord upgrade (`D+J → 2 キー出力` → `+L で BS 巻き戻し → どぅ`) と薙刀式 JSON 同梱 (commit 4ec2353)。**Phase 4.5 完遂**: 配列図盤面 (KeyboardMock) のリアルタイム押下ハイライト + router 別の動的ラベル差替 (base / Shift modifier 面 / 単一前置レイヤー / 多キー chord layer / 親指シフト面) + Right Alt セル追加で NICOLA 右親指可視化。残りは 新下駄 + 出力モード切替 UI + 「使われないキーを薄表示 (dim)」 等。OS 別分岐は置かず、ごく標準的な BT HID キーボードとして振る舞う方針
- `android-kide/` — KIDE Android プロジェクト、`com.msonrm.kide` namespace。GIME / GiDE とはバージョン独立（タグ prefix `kide-android-v*`、versionCode 環境変数 `KIDE_VERSION_CODE`）。構成:
   - `hid/` — GiDE から流用 (`HidConstants` / `KeystrokeEmitter` / `UsHidUsageTable` / `HidKeyboardManager` / `HidKeyboardService`)。`KeystrokeEmitter.send(device, usage, modifier)` と `HidKeyboardManager.sendHidUsage` / `executeAction(KeyAction)` を追加
   - `input/KeyEventSnapshot.kt` — USB HID キーボード判定 + KeyEvent → snapshot
   - `engine/EvdevToHidUsageTable.kt` — `KeyEvent.scanCode` (Linux evdev) → USB HID Usage。alphanumeric / 主要記号 / 制御 / 矢印 / F キー
   - `engine/HidModifier.kt` — `KeyEvent.metaState` → HID modifier byte (Shift / Ctrl / Alt / GUI の左右尊重、Lock 系無視)
   - `engine/SimpleKeyRemap.kt` — Identity / Dvorak ハードコード + `fromKeymap(KeymapDefinition)` で JSON 由来 (Colemak 等) を構築
   - `engine/KeyAction.kt` — sealed class: `None` / `SingleStroke` / `RollbackAndSendString` / `EmitStrokes(List<KanaStroke>)` (JIS かな出力用)。`KanaStroke(usage, modifier)` データクラスも同居
   - `engine/KeyRouter.kt` + `KeyInput` — ルーター interface。`route()` (sync API, sequential 用) と `keyDown(input, emit)` / `keyUp(input, emit)` (async-callback API, chord 用) の 2 系統。`KeyInput` は `scanCode` も持ち、Right Alt 等 HID Usage 表に無いキーを chord router が検出可能。**配列図盤面用 visualization API** (Phase 4.5): `expectedOutputLabels: Map<Byte, String>` (base 単打ラベル) + `expectedShiftedOutputLabels` (Shift 押下時の差替面) + `prefixShiftUsages: Set<Byte>` (前置シフトキーの ⇧ マーカー対象) + `prefixLayers: Map<Byte, Map<Byte, String>>` (sequential 前置レイヤー) + `multiPrefixLayers: Map<Set<Byte>, Map<Byte, String>>` (chord n キー前置レイヤー、 全 (n-1)-subset に symmetric 登録) + `chordRelevantUsages: Set<Byte>` (pressedHidUsages ∩ で 「今握ってる chord キー subset」 を計算) + `pendingPrefixUsage: StateFlow<Byte?>` (sequential router が buffer に prefix を貯めた瞬間に発火、 KeyboardMock が collectAsState で観測)
   - `engine/SimpleKeyRemapRouter.kt` — Phase 2 の SimpleKeyRemap を Router 化
   - `engine/KanaToRomajiTable.kt` — ひらがな → ASCII ローマ字 (AZIK ASCII 出力 expansion 用)。2 char kana の早期 return を廃止 (`かん` が空文字化していたバグを修正済み)
   - `engine/KanaToJisKeyTable.kt` — ひらがな → JIS HID stroke 列。Mac の「日本語 - かな入力」(ANSI 物理) を基準にした 50 音 + 濁音/半濁音 (2 stroke 合成) + 小書き (Shift+key) + 句読点 + 単独 ゛ / ゜ + ろ (backslash)。受信機 OS 別分岐なし
   - `engine/AzikRouter.kt` — sequential ASCII 出力。任意長最長一致 + eager + BS。共通プレフィックス削減 + マッチ後 buffer クリアで誤連鎖防止。`fromKeymap(def)` で JSON 由来テーブルを受け取る (旧 hardcoded `AzikTable.MAP` は削除)
   - `engine/SequentialKanaRouter.kt` — sequential JIS かな出力。**buffer + 最長一致 + 完成 kana 単位で emit** モデル (eager+rollback 不要)。`explicitKeys` (= `inputMappings` 直接書き) は `hasLongerPrefix` をスキップして即 emit (月配列 `q → そ` 1 stroke 遅延バグ修正)。AZIK / 月配列 両方を 1 つの実装で扱う
   - `engine/ChordKey.kt` — `KeyLogicKit/IME/ChordKey.swift` の Kotlin port。33 キー (QWERTY 30 + space + leftThumb + rightThumb) の bit mask (Long)
   - `engine/SimultaneousKeyBuffer.kt` — `SimultaneousKeyBuffer.swift` の Kotlin port。Phase enum (accumulating / passthrough / shiftMode) + inter-key timing (50ms window) + idle gating + シフトモード移行。chord 確定後にシフト保持中で新 letter が来たら「同じシフト + 新 letter」 の新 chord として再評価 (高速 NICOLA 連打対応)。**3 キー chord upgrade 実装済み** (commit 4ec2353): chord {D,J} 出力後 L を追加で {D,J,L} → どぅ に差し替え。直前出力 stroke 数 (`lastChordEmittedStrokeCount`) だけ BS で巻き戻して新 strokes 送出 (`replaceCount` を onOutput に渡す)。優先順位は `(a) 3 キー chord upgrade → (b) NICOLA 風 rolling (シフト保持中 chord 再構成)`
   - `engine/ChordKanaRouter.kt` — chord JSON 定義を `SimultaneousKeyBuffer` と組み合わせて消費する router。`KanaToJisKeyTable` で kana を JIS HID stroke に変換して emit。Right Alt 等の HID Usage 表に無いキーを scanCode で検出。非 chord キー (BS / Enter / 矢印 等) は buffer reset + 元 HID stroke を passthrough。**配列図 visualization** (Phase 4.5): `fromKeymap` 時に `buildChordVisualLabels` が lookupTable を 1 キー (base) と n キー (multiPrefixLayers) に分解。 n キーは各キーを 「次に押す候補」、 残りを 「握っている状態」 として全 n 通り symmetrically 登録 (`H+J+K → さ` なら `{H,J}→{K:さ}`, `{H,K}→{J:さ}`, `{J,K}→{H:さ}`)。 `resolvePhysicalKey("rightAlt")` は HID Usage 0xE6 と scanCode 0x64 の両方に dual-register (chord 入力は scanCode fallback で動き、 prefixShifts 推論は usageMap[0xE6] 経由で立つ)
   - `keymap/KeymapDefinition.kt` + `keymap/KeymapLoader.kt` + `keymap/RomajiBaseTable.kt` + `keymap/KeymapExpansion.kt` — `web/public/keymaps/*.json` (v1) を Kotlin で扱うデータクラス + `org.json` ベースのパーサ + 標準ローマ字ベーステーブル + 展開ロジック (`expandToAzikAsciiTable` (AZIK ASCII 用) と `expandToKanaTable` (JIS かな出力用) の 2 系統)。`MainActivity.buildAvailableRouters()` が起動時に `assets/keymaps/` を一括ロードし: sequential + `keyRemap` → `SimpleKeyRemapRouter` (Colemak)、sequential + `inputMappings`/`suffixRules`/`inputBase` → **`AzikRouter` (ASCII 出力) と `SequentialKanaRouter` (かな出力) を両方** (AZIK / 月配列)、chord → `ChordKanaRouter` (NICOLA)
   - `assets/keymaps/*.json` — `web/public/keymaps/` から流用 (Identity 系: `romaji_us/jis.json`, `romaji_colemak_us/jis.json` / sequential: `azik_us/jis.json`, `tsuki2-263_us/jis.json` / chord: `nicola_us/jis.json`, `naginata_us/jis.json` を同梱)。新配列を追加するときは JSON を置くだけで起動時に自動ロード対象になる
   - `MainActivity.kt` — `dispatchKeyEvent` で Tab/Enter/矢印含む全 KeyEvent 捕捉。ACTION_DOWN は `handleKeyDown` で router の `keyDown` callback 経由、ACTION_UP は `handleKeyUp` で `keyUp` callback 経由 (chord router が single tap / shift release 判定に使う)。`HID_USAGE_CAPS_LOCK` は consume せず Android に流して状態同期、letter キーには CapsLock XOR Shift を適用。送出は Send to host トグル ON 時のみ。**配列図盤面** (Phase 4.5、 `KeyboardMock` / `KeyboardKey` / `KeyboardMockRows`): US ANSI 風 5 段グリッドで現在 router の動的ラベルを表示。 `pressedHidUsages: Set<Byte>` を `updatePressedHidUsages` (KeyEvent.scanCode 経由 + 修飾子は `modifierUsageFromKeyCode` で keyCode → 0xE0-0xE7 fallback) でリアルタイム追跡し primaryContainer ハイライト。 ラベル選択は 「多キー chord layer → 単一前置 layer → Shift 面 → router base → 物理キーラベル」 の優先順。 親指行は `⇧L | ␣ | ⌥R | ⇧R` の 13 ユニット幅 (Right Alt セルで NICOLA 右親指レイヤー可視化)。 ギャップは `Arrangement.spacedBy` ではなくセル padding 方式 (`Modifier.weight + .padding(horizontal=2.dp)`) で 行毎のセル数差によるキー幅ズレを回避
- **フォローアップ: 薙刀式 judgment=mutual（相互シフト）未対応** (2026-07-18)。keymap v1 に `chordConfig.judgment`（`window`/`mutual`）が追加され、naginata JSON（assets 同梱分含む）は `"judgment": "mutual"` に切り替わったが、KIDE の `SimultaneousKeyBuffer.kt` は window 実装（inter-key timing + idle gating）のまま = **kide 上の薙刀式は従来どおり時間窓で動く**（`KeymapLoader` は未知フィールドを無視するため動作影響なし）。本家仕様は「ミリ秒を見ない相互シフト」（`docs/keymap-format.md`「judgment: 判定方式」参照）。golden の mutual 依存ケースは `skip: ["kide"]` 済み。追随する場合は Swift `SimultaneousKeyBuffer.swift` の mutual パス（heldKeys 判定 + armed 持続 + fall-through disarm）を port し、golden の skip を外す。なお BT HID 転送では単打確定が keyUp 遅延になるため、実機のレイテンシ感は要検証
