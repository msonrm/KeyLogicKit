# keymap v2 スケッチ — 判定方式の分類学とパイプライン構想

**状態: 設計メモ(未実装・未確定)**。実装計画ではなく、将来 keymap フォーマットを拡張する際の**判断の座標系**を書き留めたもの。2026-07-19 の設計討議(薙刀式 機能キー修正 PR #681 の後続議論)より。§1.5(入力情報量の階段)は 2026-07-20 の VS Code 拡張化検討、§3.6(配列とキーボードの分離)は同日の JIS/US 二重ファイル検討より追記。

## 0. 位置づけ — 「QMK ファームウェアを逆方向から実装している」

キーボード界隈には 2 つの語彙体系がある:

- **QMK/ZMK(ファームウェア界隈)**: 機構に名前を付けた。Tap-Hold、Momentary Layer、Combo、
  `TAPPING_TERM`、`PERMISSIVE_HOLD`、`HOLD_ON_OTHER_KEY_PRESS`、ZMK の "flavor"
  (`hold-preferred` / `balanced` / `tap-preferred`)。任意のキーに任意の意味を載せる一般化問題を
  ボトムアップに解いたので、**判定方針の語彙が精緻**。ただし日本語かなテーブルの意味論
  (3キー目差し替え、スパース fall-through、シフト面のかな表)は貧しく、QMK 上の薙刀式実装は
  専用 C コードになる
- **配列界隈**: 配列(作品)に名前を付けた。NICOLA、月配列、薙刀式 — 方式は作品名と癒着し、
  機構の語彙が無い。その負債の実例が薙刀式で、作者自身が「説明が面倒なので伝わりやすい
  『同時押し』という言葉を使ってきたツケ」と書き、「もし分りやすい用語があったら、発明する
  べきだ」と述べている(大岡俊彦 2026-07-11 記事、下記参考資料)。LLL 自身も当初この誤解の
  まま薙刀式を時間窓で実装していた(2026-07-18 に judgment=mutual へ修正)

keymap JSON はこの中間に立つ: **データ駆動のかなテーブル(配列界隈の資産)+ 判定方針の
宣言(ファームウェア界隈の資産)**。配列の側から出発して機構を一般化していくので、
「QMK を逆方向から実装する」形になる。目的は QMK 互換ではなく、**新配列の忠実かつ簡易な
記述言語**として keymap JSON を豊かにすること。

## 1. 分類学 — 方式は「軸の組合せ」である

### 軸 0: 打鍵の重なりを意味に使うか

**使わない = 逐次系**。完結した打鍵の列だけで意味が決まる。亜種:

| 亜種 | 例 | 機構 | QMK 対応 |
|---|---|---|---|
| 配字換え | 大西配列(ローマ字並べ替え) | keyRemap | keycode remap |
| テーブル逐次 | ローマ字、AZIK | inputMappings(greedy longest-match) | — |
| **前置**シフト | 月配列(シフトをタップ→離してから次打鍵) | inputMappings で表現済み | **One-Shot Layer** |
| **後置**変調 | JIS かな入力の濁点キー、GIME の濁点トグル/拗音後置シフト | 直前出力を変換 | — |
| 逐次系の時間窓 | ケータイのトグル打ち、GIME の「、」連打→「。」差し替え | 同キー連打の窓 | Tap Dance |

前置と後置は鏡像。時間窓は重なり系の専有物ではなく逐次系にも現れる。

**使う = 重なり系**。ここで軸 1・軸 2 が直交する。

### 軸 1(重なり系): グルーピングの判定 × 軸 2: 順序

| | 順序不問 | 順序有意 |
|---|---|---|
| **時間窓**(同時に叩く) | ② NICOLA・新下駄 | (実例なし — 下記の人間工学的理由で今後も出ない見込み) |
| **状態**(押しっぱなし=ホールド→タップ) | ③ 薙刀式「相互シフト」 | ④ 風神配列 |
| **全リリース一括**(ストローク) | ⑤ メジロ式(Plover 系ステノ) | (ステノは定義上 unordered) |

- ② が本来の意味の「同時打鍵」— 打鍵の瞬間的重なりを意味に使うので時間窓が本質的に必要
- ③④ は「ホールドシフト」— 押しっぱなしがシフト面。時間無関係。③ は順序を捨てる
  (薙刀式 v18 は全ペアを逆順二重定義 = 意図的な順序無意味化)、④ は順序を情報に使う
- ⑤ は「全部押して離した瞬間に一括解決」。現行 chord(2キー目 keyDown 早期解決+差し替え)
  とも mutual とも解決タイミングが違う第 3 の様式
- ④ の内部細分: 風神は「順序付き chord 解決」ではなく**逐次変調**(各打鍵が即確定し、
  押下集合が後続打鍵の面を変える。1 打目は巻き戻らない — 出力がローマ字で音韻合成可能
  だから成立)。仮想の「K>A=か / A>K=が」は同じ象限の chord 解決型

### 軸 3: 役割型入力(判定を不要にする方向の拡張)

判定方式(window/mutual/flavor)が必要になるのは**同じキーがタップ役とホールド役を兼ねる**
から。役割をハードウェア/宣言で分離すると曖昧性が原理的に消え、判定そのものが不要になる。

実例 = GIME ゲームパッド日本語: ホールド役(D-pad 方向 + LB = 行選択)とタップ役
(フェイスボタン = 段選択)が別部品なので、KeyRouter/判定を持たない専用リゾルバで足りる。
QMK 語では「タップ意味を持たない純粋 MO キーだけの配列」。GIME はさらに後置変調
(濁点トグル・拗音後置)、逐次時間窓(、→。差し替え)、リリース発火(LT+RT=っ)を部位ごとに
使い分けた複合系であり、「方式=軸の組合せ」の好例。

軸の座標系は複雑側(順序・ストローク)だけでなく**単純側(役割分離で判定消滅)にも延びる**。

### 1.5 分類の帰結: 方式が要求する入力情報量 — ホスト適合の判定則

各判定方式には成立に必要な**入力イベントの最低語彙**があり、ホスト環境(寄生先)が
どこまで提供するかで、動く方式の範囲が機械的に決まる(2026-07-20、hechima の
VS Code 拡張化検討より)。

入力イベント列の階段(下ほど情報が多い):

| 段 | 提供されるもの | ホストの例 |
|---|---|---|
| **L1: 文字列** | 「文字が入力された」という結果のみ。押鍵/離鍵という物理事象は消えた後 | VS Code `type` コマンド乗っ取り(native/web 共通) |
| **L2: keydown 列** | 押鍵イベント+タイムスタンプ。離鍵は見えない | keybinding 型 API 全般 |
| **L3: keydown+keyup 列** | 押下集合(heldKeys)が構成できる | DOM keydown/keyup、iOS pressesBegan/Ended、USB HID(kide — レポート自体が押下集合)、Gamepad API(状態ポーリングなので保持状態が直接手に入る) |

方式ごとの最低要求:

| 方式 | 最低段 | 理由 |
|---|---|---|
| ① 逐次(前置シフト含む) | L1 | 完結した打鍵列だけで意味が決まる(軸 0 の定義そのまま) |
| ② 時間窓 | L2(近似) | 窓判定は keydown タイムスタンプだけで書ける。ただし単打確定の契機が「次イベント or タイマー」になり、遅延 or eager+rollback を飲む |
| ③④ 状態(mutual / 順序有意) | **L3 必須** | 「押している間」が keydown〜keyup 区間で定義される。L2 以下では「D ホールドで x,y,z」と「D 単打後に x,y,z」が**同一イベント列 = 識別不能**(近似の問題ではなく情報が入力に無い) |
| ⑤ 全リリース一括 | **L3 必須** | 発火契機そのものが「全キー keyup」 |
| 役割型(軸 3) | L3 相当 | holder の保持状態が要る |

定式化: keyup は「判定パラメータの一つ」ではなく、**判定の入力である押下集合を構成する
唯一の除去イベント**。keydown だけでは集合は増える一方で、「押しっぱなし」と「離した後」の
区別が消える。SandS の単打/ホールド判別、連続シフトの終了、シフトホールドの解除は
すべて keyup の位置で定義されるため、L3 を欠くホストではこれらの意味論ごと消滅する。
現行実装もこの構造を映している: SimultaneousKeyBuffer は heldKeys を軸に、単打=全キー
リリースで出力、シフトホールド=シフトキー keyup で終了と、**確定契機の大半を keyup 側に
置いている**。

L1 固有の注意: 物理キーの同定を文字からの逆写像に頼るため、ホスト側キーボード
レイアウトの仮定(US/JIS)が必要になる。HID コードを直接取る L3(kide・KeyLogicKit)には
無い問題。

ホスト適合を検討するときは「その環境は L いくつか」を最初に問えば、動く方式の範囲が
上の表から機械的に出る(例: VS Code native エディタ = L1 → 逐次系のみ。薙刀式・新下駄を
載せるには webview 等で L3 を確保するしかない)。

## 2. v1 の現在地

| 分類 | v1 での表現 |
|---|---|
| ① 逐次(前置シフト含む) | `behavior: sequential` + inputMappings |
| ② 時間窓 | `behavior: chord` + `judgment: "window"` |
| ③ ホールド・順序不問 | `behavior: chord` + `judgment: "mutual"`(v1.3.0) |
| ④ ホールド・順序有意 | **空きマス** |
| ⑤ 全リリース一括 | **空きマス** |
| 後置変調 | **空きマス**(GIME は仕様外の専用リゾルバで実装) |
| 役割型入力 | **空きマス**(同上) |

`judgment` フィールドは ZMK の flavor の発想を配列記述フォーマットに持ち込んだもの。
window 実装の idle ゲーティングは ZMK `require-prior-idle-ms` の移植 — v1 は既に両言語の
ハイブリッドである。

## 3. v2 構想の柱

### 3.1 パイプライン化 — behavior の XOR をやめ、段の合成にする

現行の `behavior.type`(sequential XOR chord)は**混成配列を表現できない**:

- 風神: 逐次確定 + ホールド変調(どちらの型にも入らない)
- メジロ式: ストローク解決(chord 的)の出力列に辞書マッチ(sequential 的)— 2 層直列
- 月配列派生: 前置シフトと同時打鍵の併用(やまぶき等に実在)

エンジン内部には両機構が既にあり、宣言が排他にしているだけ。chord 出力は現在
`appendDirectKana` で直接かなになるが、これを sequential の inputMappings に**通せる**
ようにすると:

```
段1: グルーピング/解決層(逐次はグルーピングなし。window/mutual/stroke/変調)
      → トークン列(かな or ローマ字 or ストローク記号)
段2: 列マッチ層(inputMappings 相当の greedy longest-match)
      → かな
```

- NICOLA = 段1(window)のみ、直接かな
- 月配列 = 段2 のみ
- 風神 = 段1(ホールド変調、ローマ字出力)+ 段2(既存ローマ字テーブル)
- メジロ式 = 段1(stroke)+ 段2(ストローク列辞書)

各段は宣言的なまま合成する。「冒頭を読めば解釈体制が確定する」という v1 の美点
(同期チェック・golden・可読性)は、段ごとの宣言で維持する。

### 3.2 順序付き chord エントリ

`"K+A": "か"`(順序不問)に加えて `"K>A": "か", "A>K": "が"`(順序付き、優先マッチ)の語彙。

- 実装コストは小さい: mutual は 2 キー目 keyDown で発火する仕組み上、**押下順を既に保持
  している**(`mutualOrder` / `chordGroupOrder` — 現在は単打フォールバックの出力順にのみ使用)。
  捨てているのは lookup 直前のビットマスク化の瞬間だけ
- 後方互換: 既存 JSON は全て順序不問のまま
- **人間工学の警告(ドキュメント必須)**: 同時に叩く 2 キーの着弾順は高速打鍵で制御不能
  (重なり 10〜30ms)。順序を意味に使ってよいのは**ホールド→タップの非対称ジェスチャ**
  (どちらがホルダーか)のみ。「同時叩きの順で 2 音を選ぶ」語彙設計は配列作者が踏む罠であり、
  時間窓×順序有意のマスに実例が無いのはこの制約が配列史に現れたものと解釈できる
- 情報量: 順序 = ペアあたり +1 ビット。30 キーの 2 キーペア 435 → 870。専用シフトキーを
  増やさずに濁音/拗音系の面を順序で符号化できる可能性(30 キー内へのモーラ高密度配置)

### 3.3 全リリース一括解決(仮: `judgment: "stroke"`)

ステノ様式。全キーが離れた時点でグループ全体を 1 回解決する。

- テーブル表現の方針: メジロ式のような合成的定義(子音キー群×母音キー群の直積、実装は
  Python)は**正典を生成器に置き、交換形式は flat な lookupTable に落とす**。
  前例 = AZIK の `suffixRules`(生成的テーブルの妥協点)、index.json の自動生成
- スコープ境界(重要): 飲むのは**ストローク→モーラのかなレイヤまで**。複数ストローク辞書で
  漢字を直接出す層(TAk-SAn→沢山、活用形生成)は Plover の再実装であり keymap の管轄外。
  やるなら hechima 側に「ステノ辞書コンバータ」を挿す話(KeyLogicKit の管轄 =
  キー入力→かな解決まで、golden のスコープと同一)

### 3.4 役割型入力の宣言

キーに `role: holder | tapper | both`(仮)を宣言できると、holder 専用キーは単打意味を持たず
判定が不要になる。GIME 型の入力(ゲームパッド、フットペダル等)を仕様内に収める道であると
同時に、キーボード配列でも「専用シフトキーの宣言」を shiftKeys 上限(現行 2)より一般的に
表現し直せる。判定を複雑にする拡張ではなく**消す**拡張。

### 3.5 アクション語彙の拡張(薙刀式 編集モードの真の障害)

**編集モードの不在は behavior 宣言の不自由が原因ではない**。モード構造(D+F 両ホールドで
面を出し、押しっぱなしで連打)は mutual の連続シフト(armed 持続)+ 3 キーエントリで
**現行機構のまま表現できる**。U+I ホールドの固有名詞ショートカットに至っては
`"I+U+W": "水村"` 形式で今の lookupTable でも書ける可能性が高い(値はただの文字列)。

実際の障害は 2 つ:

1. **アクション語彙の貧しさ**: 編集モードの中身は `{Home}`、`Shift+End→BS`、`{Del 2}`
   (リピート)、`^i`、`「+改行` のような**編集コマンド列**。現行 specialActions は原子
   アクション単発のみ。必要なのはマクロ列・リピート・Home/End/選択系の語彙
2. **文書所有権の境界**: 編集モードが操作するのは確定済みテキスト = ホストの文書。
   keymap の管轄外であり、拡張すべきは hechima の `cb.hostKey` 橋(現在 Backspace と矢印
   のみ)の語彙。keymap の構造の問題ではない

> **参考(ホスト側の将来メモ): EditContext API** — W3C Web Editing WG の暫定仕様
> (Chromium 121+ のみ出荷、Firefox 肯定的・未実装、Safari 態度不明)。「アプリが文書と
> 描画を所有し、OS IME は textupdate を送るクライアントになる」— **cb 契約と構造的に
> 同一の境界を、OS IME 側から公式化したもの**(cb 契約の鏡像)。hechima ホストが採用すると:
> ① contenteditable からの解放(自前描画 + 入力配管のみ受領)② 同一文書モデルに
> hechima(cb 経由)と OS IME(EditContext 経由)を差し替え可能な入力ソースとして共存させ
> られる ③ web 縦書きエディタの現実的経路。ただし受け手側 API であり、web ページが
> IME の提供者になる道(放棄された W3C IME API の方向)は塞がったまま = hechima の
> 存在理由は変わらない。単一エンジンのため採用時は contenteditable フォールバックの
> 二重経路が必要。keymap でなく hechima ホスト統合(`docs/hechima-session-embedding.md`)
> の将来課題。

### 3.6 配列とキーボードの分離 — JIS/US 二重ファイルの解消

**現状の事実(2026-07-20、全 6 ペアを diff)**。同一配列の `_jis` / `_us` の差分は 3 類型:

| 類型 | 実例 | 差分の中身 |
|---|---|---|
| メタのみ | 薙刀式・AZIK・ローマ字・Colemak | name/description/keyboardLayout + JIS 固有キー(lang1/lang2 の modeKeys)。**かなテーブル本体は完全同一** |
| 役割バインド | NICOLA | 親指シフトの物理割当が本質的に違う(JIS = 無変換/変換、US = space/rightAlt + space に単打 convert) |
| 文字ずれ | 月配列 | JIS `@`→ち `:`→れ vs US `[`→ち `'`→れ — 同一物理キーの、OS レイアウト設定による表記差 |

原因は**定義のベースの非対称**: chord は hidToKey = 物理キー名ベースでレイアウト非依存、
sequential は inputMappings = 文字(event.characters)ベース。

ただし文字ベースはバグではない。**ローマ字系(テーブル逐次)では正しい設計**で、OS の
レイアウト設定(Colemak/Dvorak 等)をそのまま尊重できる — AZIK の JIS/US 差分がメタだけで
済んでいるのはこのため。漏れになるのは「**位置で定義された配列**(月配列 = かな配列。作者の
意図は『この物理位置=れ』であって『`:`=れ』ではない)を文字で記述している」場合だけ。
物理事象からの写像を OS に委ねた分だけ情報を失う、という §1.5 L1 の注意と同根の現象。

**v2 の因数分解**: キーボード固有部は 2 種に分かれ、置き場が違う。

1. **全配列共通のレイアウト差**(文字↔物理の対応): 各配列ファイルに入れると全配列に同じ
   重複が戻る。正しい置き場は**エンジン側の共有正規化テーブル**(1 箇所。web = e.code、
   iOS = KeyEvent.HIDKeyCode で物理は取得済み = 両実装とも可能)。配列は自分の定義ベースを
   宣言する(仮: `base: positional | characters`)— positional なら正規化を適用して 1 本化、
   characters(ローマ字系)は現状どおり OS 解釈を尊重
2. **配列固有の役割バインド**(NICOLA 親指): §3.4 の role 宣言の適用先。配列は
   「leftThumb / rightThumb が要る」と宣言し、`layouts: { jis: {…}, us: {…} }` で
   役割→物理の対応だけを書き分ける

帰結: **1 配列 1 ファイル。layouts セクションを持つのは NICOLA 型だけ**。lang1/lang2 の
modeKeys(現状 JIS 版のみ)は「物理的に存在しないキーのバインドは無視」の規則で統合できる。

注意と段取り:

- 交換形式の変更 = Swift/TS/Kotlin の 3 デコーダ + `check_keymap_sync` + golden に波及。
  単独ではやらず v2 の他の柱と同時に(ガードレール 3「宣言的なまま」は維持できる)
- 近期の中間解 = **生成器方式**(正典 1 ファイル → flat な `_jis`/`_us` を生成。前例 =
  AZIK suffixRules・index.json)。薙刀式 v18 更新の「8 ファイル同期」の痛みが再発したら導入
- ファイルを統合しても UI の JIS/US 選択は残る(物理キーボードの JIS/US をブラウザは確実に
  判定できない — Keyboard Layout API は Chromium のみ)。選択の置き場が「どのファイルを
  読むか」から「ファイル内のどのレイアウトか」に変わるだけ

---

## 4. 設計ガードレール(汎用性の維持)

1. **named mode でなく軸で増やす**。新配列が来るたびに方式名を足すのではなく、直交する軸の
   値を足す。汎用性テスト: 「この拡張は**配列名を含まない言葉で説明できるか**」
   「既存の軸の組合せとして位置づくか」。両方 Yes なら仕様候補、No なら特例(入れない)
2. **2 例目ルール**: 1 配列(1 作者ファミリー)のためだけの軸は温める。風神の behavior type
   丸ごと移植を見送ったのはこれ(順序付きエントリは軸なので候補に残る)
3. **JSON は宣言的なまま**。コード(合成規則・活用生成)は生成器側に置き、交換形式は表に落とす
4. **スコープはキー入力→かな解決まで**。変換・辞書・文書編集は hechima/ホストの領分
5. **段ごとの宣言性**でパイプラインの自由度と可読性を両取りする

## 5. 実装トリガー(いつ動くか)

- 風神の web 実験ページ(ゲームパッド方式の KeyRouter バイパス、~150 行、仕様に触れない)に
  反響が出たら → 3.1/3.2 の正式化を検討
- 雷神・大和が出揃いファミリーとして定着したら / 他作者から順序依存・ストローク系配列が
  出たら → 該当軸を実装
- 薙刀式編集モードをやりたくなったら → 3.5(アクション語彙 + hostKey 橋)から着手。
  固有名詞ショートカットは現行機構での試作から
- JIS/US 同期の痛み(v18 更新の「8 ファイル同期」の類)が再発したら → 3.6 の中間解
  (生成器方式)。v2 実装時に 3.6 の統合形式(base 宣言 + layouts)へ
