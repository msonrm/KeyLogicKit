## 1. 実験の意味 — なぜ web に縦書きエディタが無いか

web に「縦書きで**書ける**」エディタはほぼ存在しない。表示（CSS `writing-mode`）は
昔から可能なのに書く方が成立しない最大の理由は **OS の IME**で、未確定表示も候補窓も
横書きの世界の住人として縦の文章に割り込んでくる。加えて本実装で実証されたのは、
**ブラウザ（特に Safari）の縦書き編集系がキャレット・選択・再描画のレベルで壊れている**
こと（§4 のカタログ）。つまり OS IME を使う限り、IME と ブラウザ双方の壊れた縦書き
対応に挟まれて詰む。

hechima は IME がまるごとページ内にあるため、未確定文字は本文と一緒に CSS が縦に組み、
候補窓も自前 DOM で縦組にできる。ブラウザ側の壊れについても、編集・キャレット・
スクロールを全自前化することで回避可能（§5）。**EditContext なしで Safari/iPad を含む
全エンジンで縦書きエディタが成立する**という 2026-07-19 の戦略仮説は実機で裏づけられた。
iOS KanaEditor の縦書き計画（stone_engine 前提）の動機はこのページが吸収する。

### keymap v2 は不要（計画時の結論・実装で確認）

縦書きは「確定文字と未確定表示をどう描くか・候補窓をどこに出すか」= エディタ UI 層の
仕事で、キー入力→かな解決（keymap の管轄）には縦も横もない。v2 との接点は将来
「縦書き×薙刀式で編集モード」が要望として立ったときの `keymap-v2-sketch.md` §3.5
（アクション語彙 + cb.hostKey 橋）のみで、それも v1 に additive に足せる。

## 3. UX 設計判断（実機で確定済み）

いずれも既存 IME に前例の乏しい縦書き固有の設計。msonrm さんとの実機往復で決定。

1. **候補ポップアップは近接アンカー**: 注目文節の左隣（次の行方向）に縦組で出す。
   段の流れは**出た側で決める** — 左に出れば右→左、右にフリップすれば左→右。
   側の選択は空きスペース基準（左優先 → 左に入らなければ「右に入る/右の方が広い」
   場合のみ右 → 両側とも狭ければ広い方 + 画面内クランプ = 必ず見せる）。
   不変則は「**第一候補 = 注目文節のすぐ隣、Space = 文節から離れる方向**」。
   縦書きは文章が左へ進むため右フリップが稀ケースでなく、固定順だとフリップ時に
   第一候補が最遠になる。番号を消した時点で読み順の錨は無く、ハイライトが意味を運ぶ。
   前例 = OS のサブメニュー。実機評「極めて自然、混乱要素なし」
2. **番号は位置ハンドル**: 順位でなく**常に画面の左から 1,2,…** で振り、物理数字キーの
   並びと空間対応させる（rl 流では優先順と逆になるためアンカー確定後に振り直し、
   数字キー選択は位置→優先順へ逆写像）。「番号 = 座標指定、順位 = 位置と視線が運ぶ」
   という役割分担。横書き IME の「1 = 第一候補」慣習からの意図的逸脱で、実機評
   「UI として自然」
3. **矢印キーは常に視覚方向**: 変換中はセッションの論理キー（横書き前提）へ 90° 回転
   写像（物理 ↑↓ = 文節移動 / Shift = 伸縮、物理 ←→ = 候補送り。写像は描画された
   段の流れに追従）。非変換中も native に任せず `sel.modify` で統一（Safari は縦書きで
   矢印を論理方向のまま動かす。Chrome は視覚方向 — ブラウザ差も吸収）
4. **キャレットは自前描画**（横棒・点滅・`.editor-wrap` 内の相対配置 = レイアウト
   シフトに自動追従）。ポップアップは横書き時代の実装を段組だけ縦にした（フッタの
   ページ表示は横書きのまま）

## 4. Safari 縦書きバグカタログ（今回踏んだ 8 種と対処）

「web に縦書きエディタが存在しない理由」の実地記録。すべて iPad Safari 実機で確認。

| # | 事象 | 対処 |
|---|---|---|
| 1 | native キャレットを横書きメトリクスで誤描画（形状が短い・改行を跨ぐたび下へ累積ドリフト） | native は `caret-color: transparent` で隠し、ZWSP プローブ実測位置に自前 `.vcaret` を描画 |
| 2 | collapsed Range の `getBoundingClientRect` も同様に嘘をつく | 位置測定は**必ず** `measureCaretRect()`（プローブ）経由。element の rect は信用できる（ポップアップアンカーは compEl の子で OK） |
| 3 | 直交フロー（横親の中の縦ブロック）の自動サイズ計算が崩れる | コンテナに writing-mode を与えない。**横 flex + 各段に writing-mode**（縦書きタブラベルの枯れたパターン）。rl は `flex-direction: row-reverse` |
| 4 | 矢印キーが論理方向のまま（視覚方向への再写像なし） | `sel.modify` で自前写像（§3-3） |
| 5 | native 編集（BS/DEL 等）後の「キャレットを見せる」スクロールが #1 の壊れた矩形へ向かい、ページが最下部まで飛ぶ | **native 編集経路を全廃**: BS / Delete / 選択削除 / セッション素通しの 1 文字キーまで自前処理 |
| 6 | コンテンツ縮小時のスクロールクランプが次フレームで非同期に走り、同期測定がずれ、scroll イベント中の DOM 変異と衝突して選択が落ちる | scroll リスナの再描画は rAF 集約 / 編集キーは `ensureEditorSelection`（lastCaretOffset へ自動復帰）/ afterEdit は次フレーム再同期 |
| 7 | 末尾に `<br>` が無いと最終空行の行ボックスが無く、末尾境界の選択が editing host 端へ吸われる（→ DEL が永遠に no-op） | **EOF センチネル** `<br class="eof-br">` を常時維持（msonrm さん診断）。native 編集ならブラウザが自動維持するものを、全自前化に伴いホストが引き取る。末尾 collapse は `endOfTextRange()`（センチネル直前）に統一 |
| 8 | 文字削除でペイント無効化矩形を漏らし、削除済みグリフが残像として見える（スクロールで消える） | 削除系（BS/DEL・確定アンドゥ・再変換）の後に `forceTextRepaint()` = letter-spacing を 0.001px 揺らして戻す不可視の強制再描画 |

横書きページがこれらを踏まないのは、native の編集・描画がそのまま信用できるから。
**縦書きでは「表示の縦組は CSS に任せ、編集・キャレット・スクロールはホストが全部
引き取る」が唯一の成立条件**というのが本実装の結論。奇しくも hechima の cb 契約
（文書所有権はホスト）と同じ思想がエディタ内部でも徹底される形に収束した。

## 5. 実装の要点（site/src/app.ts）

- **offset 系**（docText / caretOffset / rangeAt）はテキストノードだけを歩く —
  未確定 span・EOF センチネル・プローブは自然に不可視
- `measureCaretRect()`: rangeAt でキャレット位置に ZWSP を一瞬挿して実測 → 除去 →
  normalize → `setCaretByOffset` で選択復元。**キャレット描画とスクロール追従の両方が
  これを使う**（測定系の一元化）
- `ensureEofBr()`: 編集・undo・復元・クリア後に常にセンチネルを最後尾へ（自己修復）。
  プレースホルダは `:empty` が効かなくなるため `.is-empty` クラス方式 +
  `position: absolute`（インラインのままだとプローブが文言の後ろに置かれる）
- キャレット位置の保存は自動保存に相乗り（エディタ外フォーカス時は前回値保持、
  純カーソル移動は visibilitychange の保存で拾う）
- 数字直接選択: `candWindowCount` と描画済みの流れ（`candFlowLtr`）から位置→優先順へ
  逆写像。ページング（9 件窓）は従来のまま
- **beforeinput 全ブロック（縦書きのみ）**: native 編集経路ゼロを完遂した帰結として、
  縦書きでエディタに届く `beforeinput` は定義上すべて OS 側の割り込み。cancelable な
  ものは preventDefault し、検出時は設定案内をステータス欄に一度だけ表示。
  実例 = **Android 物理キーボードのオートコレクト**（Pixel 10 実機 2026-07-21: 自動修正
  オンだと入力のローマ字を単語として挿入してくる。keydown より後の層なので keydown の
  preventDefault では防げない）。キャンセル不能な composition 系は既存の OS-IME 警告
  （compositionstart / key="Process"）が拾う

## 6. COI × Safari キャッシュ事象（縦書きと独立の副産物・サイト全体に適用済み）

検証中に発見・解体した時限爆弾。**COOP/COEP（crossOriginIsolated）サイト全体の問題**で、
薙刀式ページでも URL 直打ちで起きえた。

- 事象: 同一サイト表示中に URL 欄から再 navigation すると、COI は維持されたまま
  **キャッシュ済み応答の再利用が style / worker スクリプトだけ誤ブロック**され、
  全ページ崩壊・新規タブにも波及・キャッシュ削除まで復旧不能。
  新規ネットワーク取得は常に成功する（リロードで CSS だけ復活する事象から特定）
- 対処（多層防御、hechima repo `d12e260` 世代）:
  1. **小型アセットは全部 `Cache-Control: no-store`**（`/assets/*` + エンジン vendor 各種。
     汚染されうるキャッシュエントリ自体を無くす。wasm/辞書 2.7MB/18MB はキャッシュ維持）
  2. **worker は毎回 `?t=Date.now()`** で必ずネットワークから
  3. **インライン復旧スクリプト**を各 HTML に直接埋め込み（外部取得ゼロ = ブロック不能。
     リソース読込失敗を capture で検知 → sessionStorage ガード付きで 1 回だけ自動リロード）
  4. 診断の常設: COI 喪失・リソース読込失敗・worker ロード失敗（connectWorker が error
     イベントを見ず沈黙ハングする — labo 側 worker-client の改善候補）をステータス欄に表示
- 実機評「壊せなくなりました」
- **恒久策 = COI 非依存化（mozc wasm 単スレッド化）**。従来「寄生先（VS Code 等）の
  ための共通投資」だった項目が「主対象 iPad Safari の実害対応」に昇格
