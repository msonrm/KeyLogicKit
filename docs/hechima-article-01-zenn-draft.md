## まず、動くものから

[へちまラボ(luffa lang labo)](https://luffa-lang-labo.dev) という日本語入力の実験サイトを公開しています。ブラウザで開くだけで、こうなります。

【GIF①: トップページでローマ字入力 → 変換 → 候補選択 → 学習】

見た目はただの日本語入力ですが、**OS の IME は使っていません**。未確定文字の下線も、候補ウィンドウも、変換も、学習も、全部このページの中で動いています。変換エンジンは Mozc(Google 日本語入力の OSS 版)を WebAssembly にビルドしたもの。初回だけ辞書 18.9MB のダウンロードが走ります(体感 5 秒ほど。YouTube がそこそこ見られる回線なら同じくらいのはずです)。以後はキャッシュされ、打鍵が外部に送られることは一切ありません。

同じ仕組みで、こういうこともできます。

【GIF②: iPad + 物理キーボードで薙刀式】

同時打鍵かな配列「薙刀式」が、iPad + 物理キーボードで動いています。後で書きますが、これは OS 側の IME では原理的にできません。

【GIF③: iPhone でフリック入力】

スマホでは OS のソフトウェアキーボードを出さずに、ページが自前で描いたフリックキーボードで入力します。

この記事は、これがどうやってできているか、なぜ作ったか、の話です。整理された順番で書くと嘘になるので、実際に起きた順で書きます。**最初にあったのは PC-98 でした。**

## 発端 — VZ Editor で日本語が打てないとつまらない

【この章は一人称の物語なので、特に msonrm さんの声で書き直してほしい部分です。事実の骨組みだけ置きます】

私は [QuuBee](https://github.com/msonrm/quubee) という、PC-98 を今の環境で動かして遊ぶプロジェクト【←表現はご確認を】で、FEP を HLE(高位エミュレーション)する仕組みを作っていました。

FEP(フロントエンド・プロセッサ)というのは、DOS 時代の日本語入力の呼び名です。当時の ATOK も VJE も、OS の機能ではなく、アプリの手前に常駐するユーザーランドのソフトでした。それが後に OS へ取り込まれ、IME と呼ばれるようになった——という歴史があります。

で、PC-98 環境を現代に持ってくると、困ったことに気づきます。

- フリーで使える PC-98 用の FEP が、私の知る限り存在しない
- VZ Editor を起動しても、日本語が打てないとつまらない
- そして正直に言うと、以前から IME を自作してみたかった

そこで、実物の FEP バイナリを走らせる代わりに「FEP の顔」だけをゲスト側に見せて、中身のかな漢字変換は現代のエンジンでやる、という構成にしました。これが HLE FEP です。ブラウザで動かすので、変換エンジンもブラウザで動く必要がある。Mozc を WebAssembly にビルドすることにしました。

つまり、汎用の Web 日本語入力を作ろうとして作ったわけではありません。**PC-98 の都合で Mozc が wasm になり、気づいたら汎用の入力スタックが生えていた。**間違えた順番の物語です。

## Mozc を wasm にする — 3 つの罠

ビルドハーネスには fcitx5-mozc を使い、Emscripten(emsdk 3.1.69)でビルドします。かな列を渡すと文節と候補が JSON で返る薄い C++ ラッパーを書いてリンクする——大枠はそれだけなのですが、罠がいくつかありました。

### 罠 1: NDEBUG を揃えないと ABI が壊れる

一番痛かったのはこれです。Mozc の `candidate.h` は `#ifndef NDEBUG` のとき、候補の構造体に `std::string log` メンバを追加します。つまり **NDEBUG の有無で構造体の sizeof が変わる**。ライブラリ側は Release ビルド(-DNDEBUG)なのに、自分のラッパーだけ NDEBUG なしでコンパイルすると、同じヘッダを見ているのにメモリレイアウトが食い違って、意味不明な壊れ方をします。原因に辿り着くまでに ASan まで持ち出しました。教訓: ラッパーのコンパイルにも必ず `-DNDEBUG`。リンクスクリプトに焼き込んで、README には太字で書きました。

### 罠 2: pthreads 必須 → ページ全体に COOP/COEP

Mozc は wasm ターゲットでも pthreads を要求します(`thread.h` が pthreads なしの wasm を `#error` で弾きます)。Emscripten の pthreads は SharedArrayBuffer で実装されているので、その利用条件である cross-origin isolation——つまり **COOP/COEP ヘッダをページ全体に付ける必要**が出ます。これはホスティング制約として地味に効きます。外部埋め込み(YouTube の iframe など)が同居できなくなるからです。ラボサイトは自己完結したページなので、全面 COOP/COEP にしました。

### 罠 3: サイズとビルド時間

成果物は wasm 2.72MB + グルー JS 97.9KB + 辞書 `mozc.data` 18.9MB。辞書が支配的ですが、初回だけなので許容範囲だと思っています。CI のフルビルドは 21〜24 分(ネイティブ protoc のビルド 5 分 + ninja での wasm ビルド 15 分が支配的)。ビルドツリーをキャッシュしたら約 2 分になりました。実行速度は、GitHub Actions のランナー上で初期化 75ms。手元のブラウザでは、変換にタイムラグを感じません(体感です)。

## 層を切る、名前を付ける

wasm ができたら、その上をどう組むかです。スタックは 3 層に切りました。

| 層 | 名前 | 役割 |
|---|---|---|
| 配列エンジン | keymap-engine | キーイベント → かな。配列(ローマ字・薙刀式・NICOLA…)は JSON 定義 |
| 変換セッション層 | hechima | かな → 確定文字列。候補選択・文節伸縮・学習・確定アンドゥ・再変換の状態機械 |
| 変換エンジン | hechima-wasm | Mozc 本体。かな列 → 文節と候補 |

変換エンジンは Web Worker の中で動かし、メインスレッドとのやりとりは「かな列を渡すと文節と候補が返ってくる」変換プロトコルとして固定してあります。呼び名は「電文」。銀行の勘定系みたいですが、レトロ起源のプロジェクトなので気に入っています。境界が電文で切れているので、将来 Mozc を別のエンジンに差し替えても、上の層は無傷で済む設計です。

もうひとつの決めごとは、**UI を持ち込まない**こと。hechima は候補ウィンドウを描きません。ホスト側が実装する数個のコールバック(未確定表示・確定・キー委譲など。必須 3 つ + 省略可 7 つ)の取り決め——cb 契約と呼んでいます——へ候補データを渡すだけです。おかげで、同じ hechima の上で、ラボサイトは候補ポップアップ(スマホでは候補バー)、QuuBee は独自 UI と、まったく別の顔をしています。入力の全部品がユーザーランドにあると、UI だけ差し替えることすらできる、という実例になりました。

名前の話も少しだけ。総体の名前は hechima(へちま)です。へちまの語源は糸瓜(とうり)——「と」がいろは順で「へ」と「ち」の間に来るから「へち間」、という説があります。かな順の言葉遊びがそのまま名前になっている植物で、配列で遊ぶプロジェクトの名前として、これ以上のものは見つかりませんでした。それと、h-e-c-h-**im**-a の真ん中に IM(input method)が隠れています。なので野暮な -ime は付けていません。ラボサイトの正式名 luffa lang labo は、へちまの英名 luffa。頭文字は L.L.L. です。【脚注として由来がもうひとつ(眉村ちあき)——入れるかは判断にお任せします】

## IME の見えない仕事

作ってみて一番思い知ったのは、「日本語入力が普通に動く」に含まれている仕事の量です。変換して確定するだけなら電文一往復ですが、「普通」にはこれだけ要ります。

- 文節の伸縮(Shift+←→ で「ここではきものを」を切り直す)
- 確定アンドゥ(Ctrl+BS で直前の確定を取り消す——**学習も一緒に巻き戻す**)
- Shift+英字での英単語混在入力と、大文字/小文字/全角の綴り変換
- 確定済みテキストを選択して再変換
- ひらがな/カタカナの追加候補
- 学習・ユーザー辞書と、その永続化(OPFS。ブラウザ内に閉じていて、打鍵は外部に出ません)

こういう「見えない仕事」は、壊しても気づきにくい。なので、打鍵列と期待出力のペアを固定した回帰テスト(いわゆるゴールデンテスト)を積んでいます。現時点で変換セッション層に 82 ケース、配列エンジンに 54 ケース(この期待値は iOS の姉妹実装とも共有しています)、フリックに 20 ケース。

一度、テストが全部緑なのに、実機のキーボードからはその機能に到達できない、という事故をやりました。機能の入口を直接叩くテストばかりで、実際のキーイベント列がそこまで流れ着くかを見ていなかったのです。以来、ゴールデンテストには実打鍵シーケンスのケースを必ず混ぜています。

## 機能ツアー — 薙刀式とフリック

### 薙刀式は「同時押し」ではなかった

薙刀式は、文章を書くために設計された同時打鍵かな配列です(設計: 大岡俊彦さん)。私は最初、「同時打鍵」という言葉から「数十 ms の時間窓内に 2 キーが押されたら同時とみなす」実装をしていました。これは間違いでした。v18 の定義ファイルを読み込むと、正体は**相互シフト**——「キーを押している間だけ、他のキーの意味が変わる」という、時間ではなく状態で決まる仕組みです。押しっぱなしにすれば何文字でもシフト面が続きます(連続シフト)。大岡さんご自身が[「同時押し」という伝わりやすい言葉を使ってきたツケ、と書かれていて](https://oookaworks.seesaa.net/article/521112645.html)、言葉が実装を誤らせる例として肝に銘じました。

hechima の配列エンジンは判定方式を JSON の宣言で切り替えられます(逐次 / 時間窓 / 相互シフト)。薙刀式は `judgment: mutual` で、本家仕様どおりに動きます。

### フリックは OS キーボードを出さずに

スマホには物理キーボードがありません。では OS のソフトウェアキーボード(と OS の IME)に頼るのか——頼らないのがこのプロジェクトです。エディタに `inputmode="none"` を指定して OS キーボードを封じ、ページが自前で 12 キーフリックの盤面を描きます。盤面とフリック割当も JSON 定義です。候補表示は、縦長のポップアップだと盤面と喧嘩するので、キーボード上部の横スクロール候補バーにしました。iOS Safari のダブルタップズームとはひと悶着あって、touch-action だけでは抑えきれず、touchend の preventDefault まで必要でした。このあたりは実機検証 4 回で落ち着いています。

## 思想ひと口 — IME から FEP へ

ここまでは「どうやって」の話でした。「なぜ」を少しだけ。

日本語入力は OS の機能(IME)です。OS にあるからどのアプリでも使える——それは強みですが、裏返すと、**OS が許す範囲でしか日本語を入力できない**ということでもあります。

1. iPadOS では、物理キーボードを繋ぐとサードパーティ IME が使えません。私が iPad 用のかなエディタを自作し始めた、そもそもの理由がこれです
2. 薙刀式のような配列を OS で使うには、DvorakJ や Karabiner-Elements のような常駐ソフトが要ります。OS ごとに別物で、iPadOS にはその穴すらありません
3. ChromeOS にはサードパーティ IME 用の API(chrome.input.ime)がありましたが、「代替手段はまだ無い」と明言されたまま廃止が宣告されました([ChromeOS.dev の告知](https://chromeos.dev/en/posts/chrome-input-ime-deprecation)、[廃止 PSA](https://groups.google.com/a/chromium.org/g/chromium-extensions/c/0ybWrEVaE-I/m/8QOeRmxrBQAJ))。理由のひとつは「3 年間ずっと利用が極めて少なかったから」。不自由だから使われず、使われないから消される、という循環です

hechima は、この構図を FEP の側から迂回します。日本語入力の一式——配列も、変換も、学習も、UI も——を OS からページの中(ユーザーランド)へ引き取る。FEP という DOS 時代の言葉をわざわざ使っているのは、これが先祖返りだからです。ただし今回の地盤は、特定 OS ベンダーの拡張 API ではなく、wasm や OPFS という複数ベンダー合議の Web 標準です。

もちろんトレードオフはあります。hechima は組み込んだページでしか動きません。Gmail では使えません。IME は横に広く浅く、hechima は狭く深い。それでも、組み込んだページの中では、日本語入力のすべてに手が届きます。

**OS のものだった日本語入力を、手元に返してもらう。**——なぜそれが書き手にとって嬉しいのか、という話の続きは、長くなるので次の記事にします。

## これからと案内

実験場なので、要望駆動で増えていきます。いま考えているのは:

- **新配列の追加**。配列は JSON で書ける形式にしてあり、NICOLA・月配列・AZIK などの定義はすでに手元で動いています。載せてほしい配列があれば教えてください
- **FEP の寄生先を増やす**。エディタフレームワーク(CodeMirror / TipTap)への組み込み、Obsidian、VS Code 拡張、ブラウザ拡張——まだ構想段階ですが、層が電文で切れているので筋は見えています
- **縦書き**。OS IME に依存しないと、Web の縦書きエディタで最難関になる「未確定表示と候補ウィンドウ」が構造的に消えるので、実は相性が良いはずです
